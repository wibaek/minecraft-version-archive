# Minecraft Java Edition Release Archive Design

## Summary

Minecraft Java Edition 정식 릴리즈의 업데이트 내역과 릴리즈 날짜를 보기 좋게 정리하는 정적 사이트를 만든다. 초기 범위는 Java Edition 정식 릴리즈만 포함하며, snapshot, pre-release, release candidate, Bedrock Edition은 제외한다.

사이트는 단일 페이지 아카이브로 시작한다. 중앙 본문은 메이저 버전별 섹션을 최신순으로 배치하고, 각 메이저 버전 섹션 안에 마이너 정식 릴리즈까지 함께 정리한다. 오른쪽에는 메이저 버전 목차를 둬서 사용자가 시대별 업데이트 흐름으로 빠르게 이동할 수 있게 한다.

## Goals

- Minecraft Java Edition 정식 릴리즈의 흐름을 최신순으로 빠르게 훑을 수 있다.
- 메이저 버전 단위의 큰 변화와 마이너 업데이트의 세부 릴리즈를 한 화면 안에서 함께 볼 수 있다.
- 릴리즈 날짜, 버전명, 업데이트 이름, 핵심 변경점을 시각적으로 구분한다.
- 데이터가 페이지에 하드코딩되어도 유지보수하기 쉬운 콘텐츠 구조를 가진다.
- 정적 사이트로 배포 가능한 단순한 구조를 유지한다.

## Non-Goals

- snapshot, pre-release, release candidate 목록을 포함하지 않는다.
- Bedrock Edition 업데이트를 포함하지 않는다.
- 초기 버전에서 외부 API, 크롤러, 빌드 타임 데이터 수집을 설계하지 않는다.
- 위키처럼 모든 변경 사항을 빠짐없이 기록하지 않는다.
- 로그인, 사용자 북마크, 댓글, 서버 상태 같은 동적 기능을 만들지 않는다.

## Audience

주 사용자는 Minecraft Java Edition 업데이트 역사를 한눈에 보고 싶은 플레이어와 개발자다. 목적은 특정 버전을 깊게 조사하는 것보다, 각 메이저 버전이 어떤 시대였고 어떤 정식 릴리즈가 있었는지를 빠르게 이해하는 것이다.

## Information Architecture

페이지는 다음 순서로 구성한다.

1. Header
2. Latest release highlight
3. Major version archive
4. Right-side table of contents

### Header

Header는 사이트 이름과 짧은 설명만 포함한다. 정적 아카이브 사이트이므로 내비게이션은 최소화한다.

필수 정보:

- 사이트 이름
- Java Edition 정식 릴리즈 전용이라는 범위 표시
- 최신순 정렬이라는 힌트

### Latest Release Highlight

첫 화면에는 최신 메이저 버전 또는 최신 정식 릴리즈를 대표 영역으로 보여준다. 랜딩 페이지처럼 과하게 장식하지 않고, 이후 아카이브 섹션 일부가 첫 화면 아래에 보이게 한다.

필수 정보:

- 최신 메이저 버전
- 최신 정식 릴리즈 버전
- 릴리즈 날짜
- 업데이트 이름 또는 짧은 설명
- 핵심 변경점 3개 이내

### Major Version Archive

중앙 본문의 핵심 영역이다. 메이저 버전 섹션은 최신순으로 배치한다. 각 섹션은 메이저 버전의 큰 맥락을 먼저 보여주고, 그 아래에 정식 마이너 릴리즈를 최신순으로 나열한다.

초기 정렬 규칙:

- 메이저 버전 섹션: 최신순
- 각 메이저 버전 내부의 마이너 정식 릴리즈: 최신순

메이저 버전 섹션 필수 정보:

- 메이저 버전 번호
- 업데이트 이름
- 대표 릴리즈 날짜
- 버전 시대를 요약하는 한 문장
- 대표 변경점 3-5개
- 하위 정식 릴리즈 목록

마이너 릴리즈 항목 필수 정보:

- 버전 번호
- 릴리즈 날짜
- 업데이트 성격
- 핵심 변경점 1-3개

업데이트 성격은 자유 텍스트 대신 짧은 라벨로 표현한다.

예시 라벨:

- Major
- Content
- Fix
- Technical
- Security

### Right-Side Table of Contents

데스크톱에서는 오른쪽에 sticky TOC를 둔다. TOC는 메이저 버전만 보여주고, 마이너 릴리즈까지 모두 노출하지 않는다. 현재 보고 있는 메이저 버전은 시각적으로 강조한다.

필수 동작:

- 메이저 버전 anchor로 이동한다.
- 현재 섹션을 강조한다.
- 본문 스크롤과 독립적으로 화면에 유지된다.
- 긴 목록에서도 본문보다 더 강하게 보이지 않도록 폭과 밀도를 제한한다.

모바일에서는 오른쪽 TOC를 유지하지 않는다. 상단에 접히는 목차 또는 가로 스크롤 가능한 메이저 버전 selector로 전환한다.

## Content Model

데이터는 초기 구현에서 코드 또는 페이지 데이터 파일에 직접 하드코딩할 수 있다. 외부 수집 방식은 이번 기획 범위가 아니다.

콘텐츠 단위는 `majorRelease`와 `minorRelease`로 나눈다.

### majorRelease

필드:

- `version`: 메이저 버전 번호
- `name`: 업데이트 이름
- `releasedAt`: 대표 릴리즈 날짜
- `summary`: 버전 시대 요약
- `highlights`: 대표 변경점 목록
- `releases`: 하위 정식 릴리즈 목록

### minorRelease

필드:

- `version`: 정식 릴리즈 버전 번호
- `releasedAt`: 릴리즈 날짜
- `label`: 업데이트 성격
- `summary`: 짧은 설명
- `changes`: 핵심 변경점 목록

## Visual Direction

디자인은 Minecraft의 소재감을 참고하되, 게임 위키처럼 복잡하거나 과하게 장식적인 UI는 피한다. 정보 탐색이 핵심이므로 본문 가독성과 버전 간 구분을 우선한다.

방향:

- 전체 화면은 아카이브와 연표 사이의 느낌을 가진다.
- 메이저 버전은 큰 섹션 제목과 넓은 여백으로 시대 구분을 만든다.
- 마이너 릴리즈는 조밀하지만 읽기 쉬운 리스트나 compact card로 표현한다.
- 블록, 바이옴, 아이템을 연상시키는 색과 질감을 아주 제한적으로 사용한다.
- 날짜, 버전 번호, 변경점은 서로 다른 시각 계층을 가진다.

피해야 할 것:

- 마케팅 사이트처럼 큰 hero만 강조하는 구조
- wiki table처럼 빽빽한 표 중심 구조
- 모든 항목을 같은 카드 크기로 만드는 단조로운 그리드
- Minecraft풍 장식을 과하게 사용해 본문을 읽기 어렵게 만드는 방식

## Responsive Behavior

데스크톱:

- 본문과 오른쪽 TOC의 2열 레이아웃을 사용한다.
- 본문은 중앙에 넓게 배치하고 TOC는 보조 탐색으로 둔다.
- TOC는 sticky로 유지한다.

태블릿:

- TOC 폭을 줄이거나 본문 위쪽의 compact selector로 전환한다.
- 메이저 버전 섹션은 1열을 유지한다.

모바일:

- 단일 컬럼으로 전환한다.
- TOC는 상단 접이식 목차 또는 가로 selector로 제공한다.
- 마이너 릴리즈 항목은 날짜, 버전, 요약이 줄바꿈되어도 레이아웃이 깨지지 않아야 한다.

## Interaction

초기 버전에서 필요한 상호작용은 anchor navigation과 현재 섹션 강조 정도로 제한한다.

포함:

- TOC 클릭 시 메이저 버전 섹션으로 이동
- 현재 섹션 표시
- 모바일 목차 열기/닫기 또는 selector 이동

제외:

- 검색
- 복합 필터
- 정렬 변경
- 상세 페이지 이동
- 사용자 설정 저장

## Accessibility

- 메이저 버전 섹션은 heading 구조로 탐색 가능해야 한다.
- TOC는 실제 anchor link로 구성한다.
- 색만으로 현재 섹션을 표시하지 않는다.
- 버전 번호와 날짜는 스크린 리더에서 의미가 끊기지 않게 텍스트로 제공한다.
- 모바일 목차가 접이식이면 키보드로 열고 닫을 수 있어야 한다.

## Verification Criteria

구현 완료 기준:

- Java Edition 정식 릴리즈만 포함한다.
- 메이저 버전 섹션이 최신순으로 표시된다.
- 각 메이저 버전 안에 마이너 정식 릴리즈가 포함된다.
- 데스크톱에서 오른쪽 TOC가 메이저 버전 기준으로 동작한다.
- 모바일에서 TOC가 본문을 침범하지 않는다.
- 하드코딩된 데이터 구조가 메이저 버전과 마이너 릴리즈를 명확히 구분한다.
- 외부 데이터 수집이나 서버 기능 없이 정적 페이지로 동작한다.

## Open Decisions

현재 구현 전에 확정할 필요가 없는 항목:

- 구체적인 프론트엔드 프레임워크
- 실제 포함할 버전 범위의 시작점
- 대표 이미지 또는 텍스처 사용 여부
- 검색과 필터 기능의 추가 시점
